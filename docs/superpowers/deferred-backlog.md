- **Streaming response bodies** → **shipped** (2026-07-19, Phase-5, `feat(monitor): opt-in stream_bodies via Network.streamResourceContent`). The backlog's premise ("true chunked streaming needs a Fetch-interception path") was **factually wrong** — shipped passively instead. New opt-in `MonitorBuilder::stream_bodies(bool)` (default `false`) on `tab.monitor()` (`crates/zendriver/src/monitor/mod.rs`), reusing the monitor's existing `url_pattern` filter (no second filter). New `NetworkEvent::HttpData { request_id, chunk: Vec<u8> }`, a sibling to the pre-existing `WebSocketFrame`/`EventSourceMessage` streaming variants, emitted once per chunk (never accumulated — no memory-growth risk, unlike the whole-body path). Mechanism: CDP `Network.streamResourceContent(requestId)` — passive, no `Fetch` domain interception, no response pausing (`Fetch.takeResponseBodyAsStream` was rejected up front for exactly that reason: it forces pausing + collides with the one-actor-per-session constraint). The correlator enables streaming on `requestWillBeSent` (**not** `responseReceived`, despite that being the brief's conservative default) — empirically, against real Chrome, `responseReceived` left too little headroom: the enable call is an async CDP round-trip via a spawned task (mirroring the existing fire-and-forget `Network.enable` pattern so it never blocks the correlator's single dispatch loop), and Chrome rejects `Network.streamResourceContent` with `-32602 Request with the provided ID has already finished loading` for a fast (especially loopback) response that reaches `loadingFinished` before that call lands — reproduced directly with a 512 KiB wiremock-served body over localhost. Enabling at `requestWillBeSent` instead costs nothing (`bufferedData` on the enable reply still covers whatever arrived in the meantime) and reliably wins the race. A `streaming: HashSet<requestId>` dedups a stray repeated `requestWillBeSent` (e.g. a redirect hop reusing the same id) and is pruned on `loadingFinished`/`loadingFailed` (and cleared on `Lagged`/`Reconnected`/`Disconnected`/`Unknown` boundaries) alongside the existing `urls`/`partial` correlation state, so it can't leak. Graceful fallback: an enable-call error (pre-~124 Chrome) logs one `tracing::warn!` (shared `Arc<AtomicBool>` across every spawned enable task, so the monitor doesn't spam a warning per request once it knows streaming is unsupported) and simply never emits `HttpData` for that session — the monitor never errors out, and `NetworkExchange::body()`'s whole-body `getResponseBody` path keeps working throughout. New public `NetworkExchange::request_id()` accessor (previously `pub(crate)`-only) so a consumer — inside or outside the crate — can correlate `HttpData` chunks (which carry only `request_id`, no URL, since they arrive before the exchange completes) against the eventual `Http` exchange; without it the feature would have been uncorrelatable from outside `zendriver`. MCP: `browser_monitor_start` gained `stream_bodies: bool`; `browser_monitor_read`'s wire `MonitorEvent` gained an `http_data` variant (`request_id`, base64 `chunk_base64`, matching the existing `body_base64` convention) and its pre-existing `http` variant gained a `request_id` field (needed for the same correlation purpose on the MCP wire — without it an MCP consumer could never join `http_data` chunks to the exchange they belong to). Public API change — baseline regenerated; ledger gained entries for `NetworkEvent::HttpData`, `MonitorBuilder::stream_bodies`, `NetworkExchange::request_id`. Schema snapshots regenerated (`monitor_start_in`, `monitor_event`). Tests: unit (MockConnection) coverage of the enable call + `bufferedData`-prepend + `dataReceived`→`HttpData` decode + empty-`bufferedData` (no spurious leading chunk) + enable-error graceful fallback + `stream_bodies: false` never enabling/emitting + the `requestWillBeSent` dedup guard, all in `monitor/mod.rs`; MCP-side unit coverage of the `HttpData`→wire mapping and `stream_bodies` (de)serialization in `tools/monitor.rs`. Real-Chrome `#[ignore]` integration test (`network_monitor_http.rs::monitor_stream_bodies_reassembles_large_response`) verified locally: a 256 KiB deterministic body served by a hand-rolled slow raw-HTTP TCP listener (16 KiB writes, 20ms apart — a normal wiremock response was empirically too fast to reproduce multi-chunk delivery, see above) reassembles byte-for-byte from concatenated `HttpData` chunks, matching both `Network.getResponseBody` and the page's own `fetch()` result length. mdBook: new "Streaming bodies" section in `network-monitor.md` (mechanism, `requestWillBeSent`-trigger rationale, Chrome-version fallback) + `mcp.md` `stream_bodies` paragraph. — `crates/zendriver/src/monitor/mod.rs`, `crates/zendriver/src/monitor/events.rs`, `crates/zendriver/tests/network_monitor_http.rs`, `crates/zendriver-mcp/src/tools/monitor.rs`, `crates/zendriver-mcp/src/state.rs`, `crates/zendriver-mcp/mcp-coverage-ledger.toml`, `crates/zendriver-mcp/public-api-baseline.txt`, `docs/book/src/network-monitor.md`, `docs/book/src/mcp.md`
