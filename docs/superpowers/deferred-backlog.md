- **Random canvas strategy: two reads produce identical output** ‚Äî surfaced by the nightly `nightly-ignored-tests` job's investigation (2026-07-19); `random_canvas_differs_across_reads` is excluded from that job pending this. Root cause is NOT either of the two hypotheses the test's own comments assume (transparent-canvas fixture artifact / per-read reseed). Verified empirically on real Chrome: (1) none of the three canvas tests in `fingerprint_integration.rs` call `.stealth(StealthProfile::spoofed())`, so under the default `ProfileKind::Native` ("...No JS bootstrap") the canvas farble patch is never injected ‚Äî `.surface(Surface::Canvas, Strategy::Random)` configures a strategy nothing consumes; (2) the tests read via `Tab::evaluate()`, which runs in zendriver's isolated world (`Page.createIsolatedWorld`, world `"zendriver-eval"`) ‚Äî the stealth bootstrap patches `CanvasRenderingContext2D`/`HTMLCanvasElement` prototypes in the MAIN world only, and Chromium gives each JS world its own copy of DOM interface prototypes, so a main-world-only patch never reaches a canvas created from the isolated world; the tests need `evaluate_main()`. Fixing both moves the needle but does not make the test reliably green: a genuine **race condition** surfaced underneath it ‚Äî the first 1-3 `evaluate_main()` calls issued right after `Tab::wait_for_load()` on a freshly-navigated page consistently observe the *unpatched* native canvas API, because the stealth bootstrap script (`Page.addScriptToEvaluateOnNewDocument`) has not finished installing its monkey-patches by the time `wait_for_load()` resolves (reproduced across independent runs with differing patterns ‚Äî e.g. pristine/pristine/farbled and pristine/farbled/farbled for three sequential calls). This is a genuine stealth-effectiveness gap worth its own investigation (a real page's own early-running fingerprint script could see unspoofed values in that window), separate from the test fix. On top of that, once the patch IS confirmed active, `canvas.js`'s `farble()` draws from ONE continuously-advancing PRNG shared by the whole page (not content-hash-keyed), so two reads of *identical* pixel content get *different* noise even under `Strategy::Seeded` ‚Äî apparently contradicting `seeded_canvas_is_stable_across_reads`'s own doc comment ("the farble is deterministic per-seed, so the same pixel data produces the same noise each time"). Whether the intended design is content-keyed stable noise (an implementation bug) or advance-every-call (a stale test assumption) is an open design question. Needs: (a) a decision on the intended Seeded/Random semantics, (b) closing the bootstrap-script-installation race relative to `wait_for_load()`, (c) then fixing the three tests' `.stealth()`/`evaluate_main()` usage. ‚Äî `crates/zendriver/tests/fingerprint_integration.rs:87` (also affects `:37,62` less visibly), `crates/zendriver-stealth/src/patches/canvas.js`, `crates/zendriver-stealth/src/profile.rs:117-138` (`ProfileKind::Native` = "No JS bootstrap"), `crates/zendriver/src/tab.rs:1084-1157` (`evaluate` isolated-world vs `evaluate_main`)
